1. ğŸŒ Kubernetes Ingress
=========================================

An **Ingress** is a Kubernetes resource used to manage external access
to services inside the cluster, usually via HTTP or HTTPS.

Instead of exposing multiple services using NodePort or LoadBalancer,
Ingress provides a single entry point.

-----------------------------------------
ğŸ§© Why Ingress is Used
-----------------------------------------

- Provides HTTP/HTTPS routing
- Supports path-based and host-based routing
- Centralizes traffic management
- Enables TLS (HTTPS) termination
- Reduces cloud load balancer costs

Ingress requires an **Ingress Controller** (like NGINX, Traefik, etc.)
to actually work.

-----------------------------------------
ğŸ“˜ Example
-----------------------------------------

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-service
            port:
              number: 80

=========================================

**Annotations** are key-value pairs used to attach metadata to Kubernetes
objects.

Unlike labels:
- Annotations are NOT used for selection
- They are used for configuration and information

-----------------------------------------
ğŸ§© Use Cases
-----------------------------------------

- Ingress controller configuration
- Load balancer settings
- Monitoring and logging configs
- Tool-specific metadata
- Documentation or ownership info

-----------------------------------------
ğŸ“˜ Example
-----------------------------------------

  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /

3. ğŸ“¦ StatefulSets
=========================================

A **StatefulSet** is used to manage stateful applications in Kubernetes.

Unlike Deployments:
- Pods have stable identities
- Pod names are predictable
- Storage is persistent and tied to each pod

-----------------------------------------
ğŸ§© Key Features
-----------------------------------------

- Stable pod names (e.g., app-0, app-1)
- Stable network identity
- Persistent storage per pod
- Ordered pod creation and deletion

-----------------------------------------
ğŸ“˜ Example Use Cases
-----------------------------------------

- Databases (MySQL, PostgreSQL)
- Message queues (Kafka)
- Distributed systems (Zookeeper)
- Applications requiring stable storage

A **ConfigMap** is used to store non-sensitive configuration data
separately from application code.

This allows configuration changes without rebuilding container images.

-----------------------------------------
ğŸ§© What ConfigMaps Store
-----------------------------------------

- Environment variables
- Application config files
- Command-line arguments
- Feature flags

-----------------------------------------
ğŸ“˜ Example
-----------------------------------------

  name: app-config

  APP_ENV: production
  LOG_LEVEL: info

Used in a Pod:

- configMapRef:
    name: app-config
